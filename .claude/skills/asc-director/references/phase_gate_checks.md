# 阶段门禁检查 (Phase Gate Checks)

每个 Phase 的入口条件、出口条件和审查角色定义。

---

## Phase 0 → Phase 1 门禁

### 出口条件
- [ ] `tech_survey.md` 存在且非空
- [ ] `progress.md` 存在，标记 Phase 0 Done
- [ ] `validate_tech_survey.sh` 通过
- [ ] 用户已确认

---

## Phase 1 → Phase 2 门禁

### 入口条件
- [ ] `tech_survey.md` 存在且非空

### 出口条件
- [ ] `tech_design.md` 存在且非空
- [ ] `validate_tech_design.sh` 通过
- [ ] Context 章节已追加设计决策
- [ ] 用户已确认

---

## Phase 2 → Phase 3 门禁

### 入口条件
- [ ] `tech_design.md` 存在且非空

### 出口条件
- [ ] `plan.md` 存在且非空
- [ ] `plan.md` 包含 Feature 列表、Batch 编排表、Task 明细
- [ ] `validate_plan.sh` 通过
- [ ] Task Progress 和 Batch Progress 已同步到 `progress.md`
- [ ] 用户已确认

---

## 审查角色视角定义

### PM 审查视角

| 身份 | 核心问题 |
|:-----|:---------|
| **用户** | 我能理解这个设计吗？ |
| **测试** | 我能据此写测试用例吗？边界覆盖了吗？ |
| **产品** | 所有需求覆盖了吗？优先级对吗？ |

### 架构师审查视角

| 身份 | 核心问题 |
|:-----|:---------|
| **维护者** | 6 个月后能看懂吗？命名清晰？模块边界明确？ |
| **安全审计** | 注入风险？敏感数据加密？权限控制？ |
| **性能工程师** | N+1 查询？大列表分页？不必要的深拷贝？ |

### 结对伙伴审查视角

| 审查点 | 核心问题 |
|:-------|:---------|
| **上下文依赖** | 依赖排序正确？ |
| **签名完整** | 有伪代码签名？ |
| **AC 可执行** | 能直接验证？（不是"功能正常"而是"输入X输出Y"） |
