# 任务规划指南 (Plan Guide)

Phase 2 任务规划的详细规则：两级拆解（Feature → Task）+ Batch 编排 + 反模式警示。

---

## 前置要求：目录结构推断

规划开始前，MUST 先根据技术栈和 `tech_design.md` 推断标准目录结构。

---

## 第一级：Feature 拆分

从 `tech_design.md` 按**独立功能点**拆分 Feature。每个 Feature 是一个业务上可独立交付的功能单元。

### Feature 定义标准

| 属性 | 要求 |
|:-----|:-----|
| **独立性** | 该功能点可独立验证、独立测试 |
| **完整性** | 包含完整的 Model → Logic → UI（若有）链路 |
| **预估 LOC** | MUST 标注，用于后续 Batch 编排 |
| **依赖关系** | 标注依赖的其他 Feature（无则标 `-`） |

### Feature 列表格式

| Feature ID | 功能描述 | 预估 LOC | 依赖 Feature |
|:-----------|:---------|:---------|:-------------|
| F-01 | 评论数据模型定义 | ~150 | - |
| F-02 | 评论列表展示 | ~200 | F-01 |
| F-03 | 评论发送 | ~180 | F-01 |

## 第二级：Feature 内部拆 Task

每个 Feature 内部按现有逻辑拆原子 Task，遵循以下原则：

### 绝对原子性原则 (Absolute Atomicity)

**一个 Task ID = 一个目标文件（Class/Module）。**

- ❌ 错误: "实现 User 和 Order 的 Model"（这是两个文件）
- ✅ 正确: T-01 "User Model", T-02 "Order Model"

### 禁止打包 (No Batching)
严禁基于"简单"、"相似"或"同一目录"将多个独立的类合并为一个任务。即使它们属于同一模块，只要物理文件不同，MUST 拆分为不同 Task ID。

### 契约优先 (Contract First)
在写实现之前，MUST 先有独立的任务定义 Interface/Type。

### 正交剥离
- **IO 分离**: 将数据库操作剥离为 Repository 接口，业务逻辑只依赖接口，不依赖实现。
- **配置分离**: 不要把常量硬编码在逻辑里，拆分出独立的 Config 任务。

---

## 切分微操指南

### 认知负荷原则
- 任务描述包含"然后"、"同时"、"并且" → 立即拆分。
- 完成该任务需同时打开 3 个以上文件 → 拆分。

### 易变性原则
将「经常变化的部分」与「基本不变的部分」分离。如：核心算法（稳定）vs 参数解析（易变）。

### Happy/Sad Path 分离
逻辑极其复杂时，可将主流程与异常处理拆为两个连续任务（前提是同一文件，分步实现）。

---

## No Placeholders 原则

| 禁止写法 | 应该写成 |
|:---------|:---------|
| "TBD"、"TODO"、"待补充" | 现在就填写具体内容 |
| "参考现有实现" | `参考 src/utils/NetworkClient.swift` |
| "添加合适的错误处理" | `catch NetworkError → 返回 .failure(.networkUnavailable)` |
| "其他类似处理" | 逐条列出每个类/方法 |
| "与 T-03 类似" | 完整重复内容 |
| "编写测试" (无具体用例) | `testFilter_sensitiveWord_throwsException()` |

## 反模式警示

| 反模式 | 症状 | 修正 |
|:-------|:-----|:-----|
| 🚨 God Task | "完成整个支付流程" | 拆为 Model、API、Service、Repo |
| 🚨 Shotgun Task | "修改所有 Service 以支持新日志" | 按文件逐个拆分 |
| 🚨 Implicit Context | 隐含"顺便把数据库建了" | 显式创建独立 Schema 任务 |

## Batch 编排规则

在 Feature 和 Task 拆解完成后，按依赖关系将 Feature 编排为 Batch。核心原则：**减少碎片，保证每个 Batch 有足够代码量，同时最大化并行度。**

### 编排算法

1. 按依赖顺序遍历 Feature，将没有未完成前置依赖的 Feature 加入当前候选池
2. 候选池内贪心聚合：累加 LOC，直到 ≥ 500 封口为一个 Batch
3. 单个 Feature ≥ 500 LOC → 独占一个 Batch
4. 候选池剩余不足 500 且无更多可加 → 允许不足 500，直接封口
5. 无依赖冲突的多个 Batch 可并行执行

### 编排规则表

| 规则 | 说明 |
|:-----|:-----|
| **≥ 500 独占** | 单个 Feature ≥ 500 LOC → 独占一个 Batch |
| **< 500 聚合** | 同层级无依赖冲突的 Feature 聚合，直到 Batch 总量 ≥ 500 或无更多可聚合 |
| **末尾不足 500** | 允许，不强求凑满 |
| **聚合边界** | NEVER 跨依赖层级聚合（B 依赖 A，则 A 和 B 不能在同一 Batch） |
| **并行** | 无依赖冲突的 Batch 可并行执行 |

### Batch 编排表格式

| Batch ID | 包含 Feature | 预估总 LOC | 前置 Batch | 可并行 |
|:---------|:------------|:-----------|:-----------|:-------|
| B-01 | F-01 | ~150 | - | - |
| B-02 | F-02 + F-03 | ~380 | B-01 | 与 B-03 并行 |
| B-03 | F-04 | ~600 | B-01 | 与 B-02 并行 |
| B-04 | F-05 | ~300 | B-02 | - |

## 技能路由与负载计算

### Skill 匹配
检查环境中是否有工具/Skill 能覆盖该任务。命中 → 在任务清单中标记 `匹配 Skill` 列。

### 负载计算
```
复杂度 L = (涉及文件数) + (逻辑分支数)
代码量 LOC = 预估新增/修改行数

L ≤ 3 且 LOC < 200  →  通过，进入审查
否则                 →  驳回，返回继续拆分
```

---

## 三轮对抗性审查

**任何一个角色提出的异议（Veto）都 MUST 导致任务被驳回并修正。**

### R1: PM 审查 —— 关注"完整性与价值"
- **完整性闭环**：需求中每个名词在任务列表中都有对应？
- **隐性需求**：是否遗漏 Config、Utils、Constants 等支撑性文件？
- **验收标准**：AC 是否反映业务价值？（如"用户能收到邮件"而非"调用了函数"）
- **Feature 覆盖**：所有 Feature 是否都被分配到了 Batch？
- 🚨 若有遗漏 → **驳回，插入新任务**。

### R2: 架构师审查 —— 关注"结构与解耦"
- **语义奇点**：扫描任务描述中的"和"、"与"、"、"、"And"、"Or"。一旦发现连接了两个不同实体 → 立即拆分。
- **依赖倒置**：是否在实现 Service 之前就定义了 Interface？否则重排。
- **Batch 依赖**：Batch 编排是否违反依赖顺序？
- 🚨 若违反架构原则 → **驳回，拆分或重排**。

### R3: 结对伙伴审查 —— 关注"可执行性与细节"
- **上下文依赖**：T-05 引用了 T-02 的类型，T-02 排在前面？
- **签名完整**：每个 Task 的任务详情 MUST 包含伪代码签名。
- **AC 可执行**：验收标准能直接验证？（不是"功能正常"而是"输入 X 输出 Y"）
- **Batch 合理性**：聚合是否导致单个 Batch 上下文过载？
- 🚨 若指令模糊 → **驳回，补充细节**。

## Few-Shot 演示

### 场景 1：评论过滤系统

**输入**："实现评论系统，含 CommentModel 和两个过滤器。"

**Feature 拆分**：
- F-01: 评论数据层（Model + Repository）— ~100 LOC
- F-02: 过滤引擎（两个过滤器 + Config）— ~150 LOC

**Task 拆分 + Batch 编排**：
- F-01 + F-02 合计 ~250 LOC < 500 → 聚合为 Batch 1
- Batch 1: F-01(T-01 Model, T-02 Repo) + F-02(T-03 SensitiveFilter, T-04 SpamFilter, T-05 Config)

**审查**：
1. **R1 (PM)**："需要补充 Repository 层。" → 已在 F-01 中覆盖
2. **R2 (架构师)**："过滤器需拆开。" → T-03/T-04 已独立
3. **R3 (结对)**："AC 太模糊。" → AC 修正为"输入含敏感词 → 抛出 SensitiveException"

### 场景 2：大型迁移

**输入**："重构设置页面 + 天气 API 对接 + 通知推送。"

**Feature 拆分**：
- F-01: 设置页面重构 — ~600 LOC → 依赖: -
- F-02: 天气 API 对接 — ~300 LOC → 依赖: -
- F-03: 天气卡片 UI — ~200 LOC → 依赖: F-02
- F-04: 通知推送 — ~250 LOC → 依赖: F-02

**Batch 编排**：
- B-01: F-01（600 LOC，独占）
- B-02: F-02（300 LOC，独立于 F-01 → 可与 B-01 并行）
- B-03: F-03 + F-04（450 LOC，聚合，都依赖 B-02）

执行：B-01 ∥ B-02 → B-03 → 全量 Review

### 列定义

- **类型**: `Contract` / `Logic` / `UI` / `Config` / `Infra`
- **任务详情**: MUST 包含伪代码签名
- **视野 (Vision)**: 执行该任务时 MUST 读取的文件路径
  - **Logic 任务的视野 MUST 包含其依赖的 Interface 定义文件**
- **匹配 Skill**: 环境中可覆盖该任务的工具/Skill（如 `protobuf_gen`），无则填 `-`
- **复杂度**: `L`（≤2 文件, 纯机械）/ `M`（3-4 文件, 含逻辑判断）/ `H`（≥5 文件, 需架构决策）
- **验收标准**: 输入输出的边界条件和错误处理要求
