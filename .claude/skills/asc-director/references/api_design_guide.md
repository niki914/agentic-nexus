# 架构设计指南 (API Design Guide)

Phase 1 架构设计的详细工作流和输出模板。

---

## 架构特征分析清单

| 维度 | 提取内容 | 示例 |
|:-----|:---------|:-----|
| 基础设施 | 识别封装库 | `NetworkClient`、`LogUtil` |
| 安全与工具 | 强制性辅助方法 | `safeGet()`、`bd_safeObjectAtIndex` |
| 架构风格 | 命名规范和架构模式 | 驼峰 vs 下划线、MVVM / MVI / VIPER |

> MUST 复用已识别的基础设施。NEVER 重新发明轮子。

## 轻量化约束（Phase 1）
1. 优先选"最小扩展点落位"方案：能通过已有类型新增字段/新增方法解决时，禁止新增中间层与分发器。
2. "项目目录结构"章节必须写"最小改动集"，明确哪些文件不会动，避免设计范围暗中扩大。

---

## 设计隔离原则

| 测试 | 问题 | 不通过的信号 |
|:-----|:-----|:-------------|
| **What** | 它做什么？能用一句话说清吗？ | 描述中出现"和"/"同时"/"还负责" |
| **How** | 怎么使用它？接口是否清晰？ | 调用方需要了解内部实现细节 |
| **Depends** | 它依赖什么？依赖是否最小化？ | 依赖列表 > 5 个，或存在循环依赖 |

### 隔离检查 Filled Example

```
模块: SensitiveWordFilter
- What: 检测文本中是否包含敏感词，超过阈值则拒绝 → ✅ 一句话说清
- How: filter(text: string) -> FilterResult → ✅ 接口清晰
- Depends: [IFilter, FilterConfig] → ✅ 仅 2 个依赖
→ 隔离验证通过

模块: CommentService
- What: 处理评论的创建、过滤和存储 → ⚠️ 出现"和"，多职责
- How: createComment(text, author) -> Comment → ✅ 接口清晰
- Depends: [IFilter, CommentRepo, UserService, NotificationService, AuditLog] → ❌ 5 个依赖
→ 建议拆分：CommentCreator + CommentModerator
```

## 🧠 设计讨论模板

架构草案完成后，对拿不准的设计争议点 MUST 用结构化方案对比咨询用户，NEVER 用纯文本列表罗列。

**决策边界**——什么需要讨论 vs AI 自决：

| 需要讨论（MUST 列出） | AI 自决（SHOULD NOT 列出） |
|:----------------------|:--------------------------|
| 模块职责划分有多种合理方式 | 纯代码风格/命名 |
| 公共接口/协议设计 | 内部私有实现细节 |
| 有性能/复杂度/兼容性 trade-off | 已有明确先例的重复模式 |
| 需要修改公共模块 | 仅影响新增代码的决策 |

**输出格式**：

> 📐 **架构草案已完成，以下设计争议点需要你确认：**
>
> **整体方案概要**：[3-5 句话]
>
> **争议 1**：[问题描述，含背景和为什么有争议]
>
> **选项 A：[方案名]（推荐）**
> [具体描述：类/模块怎么划分、接口怎么设计]
> → 优势：… / 风险：…
>
> **选项 B：[方案名]**
> [具体描述：类/模块怎么划分、接口怎么设计]
> → 优势：… / 风险：…
>
> 💡 **我的倾向**：选项 A，因为 [基于架构特征分析和代码现状的具体理由]
>
> _(最多 3 个争议点)_

**规则**：
- 争议点 ≤ 3，只列真正有分歧的设计选择。
- 每个争议点 MUST ≥ 2 个选项，每个选项 MUST 有**方案名 + 具体描述 + 优势 + 风险**。
- MUST 给出 AI 倾向及**具体理由**。

## 红队双重审查协议

### 审查者 1：PM
- API 是否覆盖每条功能？边缘情况？
- **行动**：功能遗漏 → 补充 API。

### 审查者 2：架构师
- 是否复用了现有基础类？设计复杂度合理？
- 每模块通过 What/How/Depends？
- **行动**：重构对齐技术现状。

---

## 硬性约束

1. **零模糊**：完整签名。❌ "添加处理登录的方法" ✅ `func loginUser(username: String) -> Promise<UserSession>`
2. **上下文忠诚**：复用代码库工具类。
3. **可追溯性**：每个 API 变更追溯到需求。
4. **设计隔离**：What/How/Depends 三问检查。
